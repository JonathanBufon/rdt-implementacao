config_loader.py
Leitura dos arquivos de configuração em memória
def load_routers_config(path="roteador.config"):
    routers = {}
    with open(path) as f:
        for line in f:
            parts = line.strip().split()
            router_id = int(parts[0])
            routers[router_id] = {
                "ip":   parts[2],
                "port": int(parts[1])
            }
    return routers

def load_links_config(path="enlaces.config"):
    links = []
    with open(path) as f:
        for line in f:
            a, b, cost = [int(x) for x in line.split()]
            links.append((a, b, cost))
    return links

graph.py — Dijkstra
Caminho de menor custo e tabela de encaminhamento
import heapq

def _dijkstra(graph, source):
    dist = {source: 0}
    prev = {}
    heap = [(0, source)]
    while heap:
        d, u = heapq.heappop(heap)
        if d > dist.get(u, float("inf")):
            continue          # nó obsoleto
        for v, w in graph.get(u, []):
            nd = d + w
            if nd < dist.get(v, float("inf")):
                dist[v] = nd
                prev[v] = u        # rastreia predecessor
                heapq.heappush(heap, (nd, v))
    return dist, prev

def build_forwarding_table(graph, source):
    _, prev = _dijkstra(graph, source)
    table = {}
    for dest in graph:
        if dest == source: continue
        node = dest
        while prev.get(node) != source:
            node = prev[node]   # anda para trás até source
        table[dest] = node      # primeiro salto após source
    return table

Como funciona o Dijkstra

  • heap min-priority garante processar sempre o nó de menor custo acumulado
  • prev[] armazena o predecessor de cada nó no caminho ótimo
  • Nós obsoletos no heap são descartados pela verificação d > dist[u]
  • Complexidade: O((V + E) log V)

Como build_forwarding_table funciona

Para cada destino, caminha de volta pelo prev[] até encontrar o nó cujo predecessor é a própria origem. Esse nó é o próximo salto.

Propriedade chave

Cada roteador só conhece o próximo salto, não o caminho completo. O pacote avança hop-a-hop, e cada nó consulta sua própria tabela para decidir para onde enviar.

packet.py — Formato dos Pacotes
Serialização JSON sobre UDP

Pacote DATA

{
  "type":        "DATA",
  "seq":         1,       ← número de sequência
  "origin":      1,       ← quem originou
  "destination": 5,       ← destino final
  "payload":     "Olá",   ← até 100 chars
  "path":        [1,3,5]  ← informacional
}

Pacote ACK

{
  "type":        "ACK",
  "seq":         1,  ← mesmo seq do DATA
  "origin":      5,  ← quem envia o ACK
  "destination": 1   ← quem deve receber
}
def serialize(packet):
    return json.dumps(packet).encode()

def deserialize(data):
    return json.loads(data.decode())

Decisões de design

  • JSON — legível, fácil de inspecionar nos logs, sem dependências
  • Campo path é apenas informacional — o roteamento usa a forwarding table local, nunca o path do pacote
  • ACK fim-a-fim — o ACK viaja do destino final de volta à origem, hop-a-hop, pelo mesmo mecanismo de encaminhamento
  • seq no ACK deve corresponder ao seq do DATA (RDT 2.2 — sem NAK)

Por que SOCK_DGRAM?

UDP (SOCK_DGRAM) não tem conexão, não garante entrega nem ordem. Toda a confiabilidade é implementada manualmente pelo projeto — é esse o objetivo do trabalho.

router.py — Arquitetura de Threads
Thread principal (comandos) + Thread listener (escuta UDP)
Thread Principal — command_loop()
def command_loop(self):
  while self.running:
    try:
      cmd = input()    # bloqueia aqui
    except EOFError:
      break           # stdin fechado
    self.handle_command(cmd)
    # send_message() também bloqueia aqui
    # enquanto aguarda ACK (stop-and-wait)
  • Lê o comando send <dest> <msg>
  • Bloqueia em saw.wait_for_ack() durante envio
  • EOFError → para de aceitar comandos mas mantém o roteador vivo
Thread Listener — listen() — daemon
def start(self):
  self.running = True
  t = threading.Thread(
    target=self.listen, daemon=True)
  t.start()

def listen(self):
  while self.running:
    data, _ = self.sock.recvfrom(4096)
    pkt = deserialize(data)
    self.handle_packet(pkt)
  • daemon=True → morre junto com a thread principal
  • Processa DATA (encaminhar ou entregar) e ACK (encaminhar ou sinalizar)
  • Quando recebe ACK correto, chama saw.on_ack() que desbloqueia a thread principal

Thread safety: A única variável compartilhada entre as duas threads é o objeto StopAndWait, que usa threading.Lock + threading.Event para sincronização sem race condition.

router.py — send_message
Loop de stop-and-wait com timeout e retransmissão
def send_message(self, destination, payload):
  path = shortest_path(..., destination)
  next_hop = self.forwarding_table[destination]

  self.seq += 1
  seq = self.seq
  packet = make_data(seq, self.id,
                       destination, payload, path)

  self.saw.set_pending(seq)  # arma o timer
  attempt = 0

  while True:          # reenvio indefinido
    attempt += 1
    if attempt == 1:
      self.logger.info("[ENVIADA] ...")
    else:
      self.logger.info("[REENVIO] ...")

    self.send_packet(next_hop, packet)

    if self.saw.wait_for_ack(timeout=3):
      self.logger.info("[ACK_RECEBIDO] ...")
      break           # sucesso
    # timeout → próxima iteração reenvia

Por que Event.wait(timeout=3) e não sleep(3)?

Com Event.wait, o ACK desbloqueia o método imediatamente ao chegar — sem esperar os 3 segundos. Com sleep, a origem esperaria sempre 3s, mesmo quando o ACK chegou em 50ms.

Estados possíveis após wait_for_ack

  • True — ACK correto chegou → sai do loop
  • False — timeout de 3s → loga [REENVIO], reenvia o mesmo pacote com o mesmo seq

Escopo do stop-and-wait

Um único pending_seq por roteador. O roteador não aceita novo comando enquanto aguarda ACK da mensagem atual — command_loop está bloqueado em send_message.

LOSS_PROBABILITY = 0.10

def handle_packet(self, packet):

  seq    = packet.get("seq")
  origin = packet.get("origin")

Onde ocorre o descarte

Em qualquer roteador que receba o pacote — incluindo intermediários. Isso simula perda em qualquer enlace da rede, não só na origem ou destino.

Aplica-se a DATA e ACK

O descarte acontece antes do if ptype ==. Isso cobre o caso do slide 3-64 do Kurose/Ross: o ACK de retorno pode se perder, forçando o destino a reenviar o ACK quando receber o DATA duplicado.

router.py — Encaminhamento hop-a-hop
_handle_data e _handle_ack — cada roteador decide o próximo salto

_handle_data — pacote DATA chegou

def _handle_data(self, packet):
  dest = packet["destination"]

  if dest == self.id:
    self._deliver_packet(packet)  # chegou!
    return

  next_hop = self.forwarding_table.get(dest)
  self.send_packet(next_hop, packet)
  self.logger.info("[ENCAMINHADA] ...")

_handle_ack — ACK de retorno

def _handle_ack(self, packet):
  dest = packet["destination"]  # origem do DATA

  if dest == self.id:
    self.saw.on_ack(packet["seq"])  # desbloqueia!
    return

  next_hop = self.forwarding_table.get(dest)
  self.send_packet(next_hop, packet)

ACK é roteado como qualquer pacote

O campo destination do ACK é a origem do DATA. Cada roteador intermediário consulta forwarding_table[destination] para encaminhar o ACK de volta.

Exemplo: ACK de R5 → R1

  • R5 envia ACK com dest=1 → next_hop=R3
  • R3 recebe ACK com dest=1 → next_hop=R1
  • R1 recebe ACK com dest=1 == self.id → saw.on_ack(seq)

send_packet — o primitivo de envio

def send_packet(self, next_hop, packet):

    serialize(packet),
    (info["ip"], info["port"]))
router.py — Entrega e Deduplicação
_deliver_packet garante que a mensagem é entregue exatamente uma vez
def _deliver_packet(self, packet):
  origin  = packet["origin"]
  seq     = packet["seq"]
  payload = packet["payload"]

  # ── Deduplicação ────────────────────────
  if self.last_received_seq.get(origin) == seq:
    self._send_ack(seq, origin)  # reenvia ACK
    return   # NÃO entrega de novo

  # ── Entrega ──────────────────────────────
  self.last_received_seq[origin] = seq
  self.logger.info('[RECEBIDA] Seq %s ...', seq)
  print(f'Roteador {self.id} recebeu ...')

  self._send_ack(seq, origin)

def _send_ack(self, seq, origin):
  ack_next_hop = self.forwarding_table[origin]
  ack = make_ack(seq, self.id, origin)
  self.send_packet(ack_next_hop, ack)
  self.logger.info("[ACK_ENVIADO] ...")

Por que a deduplicação é necessária?

Se o ACK se perder, a origem faz retransmissão. O destino recebe o mesmo DATA pela segunda vez. Sem deduplicação, entregaria a mesma mensagem duas vezes ao usuário.

Como funciona

  • last_received_seq[origin] guarda o último seq entregue por cada origem
  • Se o seq chegou de novo → ACK duplicado perdido → reenvia ACK sem entregar
  • Se o seq é novo → entrega e atualiza last_received_seq

ACK separado do fluxo de entrega

_send_ack é chamado em ambos os casos (duplicata e entrega nova), garantindo que a origem sempre receba um ACK e saia do loop de retransmissão.

reliability.py — Stop-and-Wait
Sincronização entre thread principal e thread listener via threading.Event
import threading

class StopAndWait:
  def __init__(self):
    self.pending_seq = None
    self.lock      = threading.Lock()
    self.ack_event = threading.Event()

  def set_pending(self, seq):
    with self.lock:
      self.pending_seq = seq
      self.ack_event.clear()   # arma o evento

  def on_ack(self, seq):        # chamado pelo listener
    with self.lock:
      if self.pending_seq == seq:
        self.pending_seq = None
        self.ack_event.set()   # sinaliza!
        return True
    return False

  def wait_for_ack(self, timeout=3):  # bloqueante
    return self.ack_event.wait(timeout=timeout)

threading.Event vs sleep

AbordagemACK em 50msTimeout 3s
Event.wait(3)retorna em 50msretorna em 3s
sleep(3)espera 3s mesmo assimespera 3s

threading.Lock — proteção da race condition

Tanto set_pending (thread principal) quanto on_ack (thread listener) acessam pending_seq. O Lock garante que nunca sejam executados simultaneamente.

ACK de seq errado é ignorado

on_ack só sinaliza se pending_seq == seq. ACKs de mensagens antigas (que podem chegar atrasados na rede) não interferem.
